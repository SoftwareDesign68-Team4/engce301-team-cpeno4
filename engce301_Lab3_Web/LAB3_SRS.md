# SRS – ENGCE301 Lab Submission Portal

## 1. Introduction

### 1.1 Purpose (วัตถุประสงค์)
เอกสาร SRS ฉบับนี้กำหนดความต้องการของระบบ **Event Access System**  
เพื่อใช้เป็นกรณีศึกษาในวิชา **ENGCE301 – Software Design & Development** โดยมีเป้าหมาย:

- ให้นักศึกษาฝึกอ่าน/วิเคราะห์ SRS
- แปลง SRS → User Stories → Use Cases
- นำ Use Cases ไปออกแบบและพัฒนา **หน้าเว็บจริง (HTML/CSS)** ใน LAB Week 3

### 1.2 Scope (ขอบเขต)
ระบบ Event Access System จะช่วย:

- นักศึกษา:  
  - สามารถจัดกิจกรรมขึ้นเองตามความชอบของนักศึกษา  
  - สามารถเช็คอินการเข้าร่วมกิจกรรมเพื่อนเป็นหลักฐานได้ 
  - สามารถเช็คประวัติการเข้าร่วมกิจกรรมของนักศึกษาได้
- ครูฝ่ายกิจกรรม / ประธานกิจกรรม:  
  - สามารถดูกิจกรรมที่ถูกยื่นเสนอขอให้จัด
  - สามารถสั่ง อนุมัติ / แก้ไข กิจกรรมที่ถูกเสนอขึ้นมาได้  
  - Feedback กิจกรรมที่อนุมัติให้อยู่ในกฏระเบียบ

scope ของ SRS นี้เน้นฟังก์ชันหลักสำหรับวิชานี้ **ยังไม่ครอบคลุม** ระบบ login จริง, การ integrate กับระบบมหาวิทยาลัย ฯลฯ (ถือเป็น out-of-scope ในวิชานี้)

### 1.3 Actors & Roles

- **Student**  
  - ยื่น/เสนอ กิจกรรม, เช็คอินการเข้าร่วมกิจกรรม, ดูประวัติและชั่วโมงการเข้าร่วม
- **Instructor** (ครูฝ่ายกิจกรรม / ประธานกิจกรรม)  
  - ดูกิจกรรมที่ถูกยื่นเสนอ, สั่ง อนุมัติ / แก้ไข กิจกรรมที่ถูกเสนอ, Feedback กิจกรรมให้อยู่ในกฏระเบียบ
- **Teaching Assistant (TA)** (อาจมี/ไม่มีก็ได้)  
  - ดูกิจกรรมที่ถูกยื่นเสนอ, สั่ง อนุมัติ / แก้ไข กิจกรรมที่ถูกเสนอ, Feedback กิจกรรมให้อยู่ในกฏระเบียบ

### 1.4 Definitions, Acronyms
- **Access Event** – กรอกแบบฟอร์มการเสนอ/ยื่นกิจกรรม
- **Check In Event** – ลงชื่อเข้าร่วมกิจกรรมในระบบไว้
- **History Event** – ดูประวัติการเข้าร่วมกิจกรรม
- **Event Submission** – สำหรับให้ผู้มีสิทธิอนุมัติจัดกิจกรรมดูและตรวจสอบ

## 2. Overall Description

### 2.1 System Overview
ระบบเป็นเว็บแอปพลิเคชันที่แสดงรายการ Event ของวิชา ENGCE301,  
อนุญาตให้นักศึกษายื่นเสนอกิจกรรมที่ชอบ, ดูสถานะการยื่น, และให้อาจารย์/ประธานอนุมัติ–กำหนดขอบเขตผ่านระบบเดียวกัน

สำหรับวิชานี้ นักศึกษาจะสร้าง **หน้าเว็บต้นแบบ (Prototype UI)**  
เช่น Landing Page, My Labs Page, About Page โดยใช้ HTML/CSS เท่านั้น (ยังไม่ต้องมี backend จริง)

### 2.2 Assumptions
- ผู้ใช้ใช้เว็บเบราว์เซอร์สมัยใหม่ (Chrome, Edge, Firefox, ฯลฯ)
- ระบบจริงในอนาคตอาจต้องเชื่อมต่อกับระบบบัญชีมหาวิทยาลัย (Single Sign-On)  
  แต่ในรายวิชานี้ถือว่า “mock login / assume already logged in”

## 3. Functional Requirements (FR)

> *หมายเหตุ*: แต่ละ FR สามารถ map ไปเป็น Use Case ได้ 1 ตัว หรือมากกว่า

### FR-1 index Event
**Description**  
ระบบต้องแสดงฟังชั่นต่าง ๆ

**Details**
- แสดง: หัวข้อของแต่ละฟังชั่น เช่น Access Event, Check in Event
- สามารถกดเข้าใช้งานได้ทุกฟังชั่น

---

### FR-2 Access Event 
**Description**  
ระบบต้องแสดงรายการ Event ทั้งหมดที่นักศึกษายื่นเสนอเข้ามา

**Details**
- แสดง: รหัส/ชื่อ ชั้นปี, คำอธิบายกิจกรรมที่จะจัด, วัน–เวลาและสถานที่ , สถานะกิจกรรม (pass/Didn't pass)
- สามารถเรียงตาม “ลำดับการยื่นเสนอ” หรือ “กิจกรรมที่ผ่าน” ได้ (อย่างน้อย 1 แบบ)

---

### FR-3 Check In Event 
**Description**  
นักศึกษาสามารถเช็คอินภายในกิจกรรมพร้อมอัพรูปภาพเป็นหลักฐานการเข้าร่วม

**Details**
- รองรับไฟล์ `.png`, `.heic`, `.jpg` อย่างน้อย 1 ชนิด
- ไฟล์รูปขนาดไม่เกิน 300KB
- บันทึกเวลาการเช็คอิน (timestamp)

---

### FR-4 History Event 
**Description**  
นักศึกษาสามารถดูการประวัติการเข้าร่วมกิจกรรมต่างๆและชั่วโมงที่ได้รับ

**Details (แนะนำ)**  

- สามารถดูตารางกิจกรรมที่เคยเข้าร่วม
- มีรายชั่วโมงที่ได้รับต่อท้ายชื่อกิจกรรมนั้นๆ และด้ายท้ายกิจกรรมจะสรุปชั่วโมงกิจกรรมที่เข้าร่วมทั้งหมด

---

### FR-5 Event Submission
**Description**  
อาจารย์ฝ่ายกิจกรรม/ประธานกิจกรรม เท่านั้นที่มีฟังชั่นนี้

**Details**
- ดูรายละเอียดแบบฟอร์มกิจกรรมที่ถูกเสนอยื่นเข้ามา
- มีสิทธิอนุมัติหรือไม่อนุมัติให้จัดกิจกรรมที่ถูกเสนอยื่นเข้ามา
- กิจกรรมไหนที่อนุมัติแล้ว สามารถ feedback เพื่อกำหนดขอบเขตกิจกรรมนั้นได้

## 4. Non-Functional Requirements (NFR)

### NFR-1 Usability
- UI ต้องอ่านง่าย เหมาะกับการใช้งานบนจอ Desktop/Laptop สำหรับการเรียน
- ใช้ layout ที่สม่ำเสมอ เช่น header, navigation bar, content area

### NFR-2 Performance
- หน้าแสดงรายการกิจกรรม ควรโหลดภายใน ~3 วินาที (สำหรับ prototype ให้เน้นไม่ใส่ของหนักเกิน เช่น รูปภาพใหญ่ ๆ)

### NFR-3 Security (ระดับวิชาเรียน)
- ในวิชานี้ ถือว่ายังไม่ต้องทำระบบ login จริง แต่ออกแบบ UI ให้มีที่สำหรับแสดงชื่อผู้ใช้ / บทบาท  
  เพื่อเตรียมต่อยอดไปสู่ระบบจริงได้ในอนาคต

### NFR-4 Maintainability
- HTML/CSS ต้องถูกแยกไฟล์ชัดเจน (`index.html`, `styles.css` ฯลฯ)
- ใช้ class / id ที่สื่อความหมาย เพื่อให้อ่าน/แก้ไขโค้ดง่าย
